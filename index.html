  • BuildBro
    A very simple build tool with a lisp-like syntax & macros.
    BuildBro was one of my first Scala projects that I used to help teach myself the language. I'm a huge fan of Scheme, so I thought I'd base the build language on that. So in addition to having some sweet syntax, you can use simple macros to eliminate repetitive sections of your build definition. Notable parts of the implementation include a concise parser using Scala's built-in DSL, quasiquotation using the visitor pattern, a recursive dependency resolution algorithm, and a very clean main method.
  • Headsheet
    The Haskell spreadsheet.
    Headsheet was born out of my desire to use this crazy Haskell combinator called loeb: 
    loeb :: Functor f => f (f a -> a) -> f a
loeb x = fmap ($ loeb x) x -- Where the magic happens
    loeb basically acts as a "spreadsheet function". In the list case (it works for any Functor) you pass it a list of functions of type [a] -> a and it returns a result of type [a]. The crazy thing is that the [a] passed to the functions is actually the "evaluated" list itself! You'd think this would result in infinite recursion, but Haskell's lazy evaluation performs a type of dependency management that evaluates everything in the right order. Here are a few examples using loeb: 
    Prelude Data.List> loeb [\xs -> xs !! 1 + 3, const 2]
[5,2]
Prelude Data.List> loeb [const "hello ", \xs -> xs !! 0 ++ "world"] !! 1
"hello world"
    As you can see in the screenshot, the expression language for the (web-based) spreadsheet is Haskell itself. The way Headsheet works is by constructing functions for each cell that call the Haskell interpreter on the contents of that cell -- these functions take an Array and return the result of evaluating that cell (by, for example, indexing into that array and manipulating other values in the spreadsheet). Other interesting parts of the implementation include a mini web framework based on Network.HTTP.Server that handles routing and some ugly use of unsafePerformIO.
  • Spaceman
    A dead-simple in-memory spatial cache.
    Spaceman (short for "space manager") associates labels with points in space and allows you to retrieve all of the points in a specified area, as well as delete points by label. That's it. I wrote it for another project (location-based bulletin boards) so that we could cache locations and retrieve them over areas very quickly (not that we needed this kind of scalability, but writing Spaceman was fun). You interact with Spaceman over TCP, by sending simple plaintext commands. The implementation includes an immutable and expandable quadtree, QuickCheck tests for said quadtree (love me some QuickCheck), and a Parsec parser for the command language.
  • Privy
    Anonymous location-based bulletin boards.
    Privy was built with my friend hsc using Django. It allows you to create circular location-based anonymous bulletin boards. The idea is that you can make a bulletin board for your house, your school, or even your city by creating a circle around that area -- and then "share the world's secrets". The implementation is pretty complete, and I even ran an instance for a while for my friends, but its abandoned at this point.

  • CSE490h: Distributed Systems Final Project
    A distributed cache-coherent filesystem.

    Distributed systems was one of the hardest classes I've ever taken, under Tom Anderson. My partner was the immensely talented Colin Scott, now a grad student at UC Berkeley.

    For our class-long project, we built a distributed cache-coherent filesystem. That means files can be stored across multiple nodes consistently, with transactional writes and reads. We implemented multi-paxos, which allowed our system to decide upon an order for transactions as they were submitted. We also implemented a reliable in-order transport protocol from scratch that would preserve sequence numbers across node restarts. Everything ran inside a simulator (that could simulate packet drops and reorderings and such) provided by the instructor. We also used Google protobuffers for our upper layer protocol.

    Our proudest detail of the code is the elegant use of abstractions to control complexity. For example, we had a class called a Handle that essentially acted as a future so we could handle asynchronous operations easily. We also implemented a lock to operate within the context of the simulator. We built a state machine to handle the complex state of the transaction server. One of the most complicated parts of the code is the multi-paxos implementation. We structured the code so that agents could have many (generic) roles: learner, monitor, and acceptor -- corresponding to the roles described in the original paper. Messages to each agent were dispatched to the appropriate role using a demultiplexer operating according to rules in a dispatch table. Throughout the project, we used a python script I wrote to compare relative line counts for each file. If a file got too long (comparatively), we tried to refactor it.

    Another useful bit was an advanced packet tracing framework that allowed us to view the packets (in glorious color) as they were transmitted.

    For more implementation details, feel free to check out our detailed writeups in this folder.
